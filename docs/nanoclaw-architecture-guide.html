01 / 项目概览

NanoClaw 是什么

NanoClaw 是一个单进程 Node.js 应用，让你通过飞书、Telegram、Discord、Slack、WhatsApp 等聊天平台跟 Claude 对话。它不是简单的消息转发——每个对话都有独立的容器、文件系统、记忆和工具。

💬

多频道接入

飞书、Telegram、Discord、Slack、WhatsApp、iMessage、邮件……通过适配器模式统一接入

🐳

容器隔离

每个 Agent Group 运行在独立 Docker 容器里，文件系统、工具、记忆完全隔离

🧠

Claude Agent SDK

容器内运行 Claude Agent SDK，支持 Bash、文件操作、MCP 工具等完整能力

🔒

凭证安全

容器永远看不到 API Key，通过本地代理注入凭证，OneCLI 管理密钥

核心设计哲学

消息路由（Host）与 Agent 执行（Container）彻底分离。Host 只管收发消息和调度，Container 只管思考和行动，两者通过 SQLite DB 跨 mount 通信。

为什么用 SQLite 而不是 HTTP 通信？

容器通过 VirtioFS 挂载宿主机目录。SQLite 跨 mount 写入需要独占——用两个 DB 文件（inbound.db + outbound.db），Host 和 Container 各写各的，互不冲突。HTTP 需要额外端口和网络配置，SQLite 更简单可靠。

03 / 频道适配器

Channel Adapter 模式

每个聊天平台是一个频道适配器，遵循统一的 ChannelAdapter 接口，在启动时自注册到 registry。

src/channels/index.ts  (barrel file)
  import './cli.js'
  import './feishu.js'       ← 每个 import 触发 registerChannelAdapter()
  import './telegram.js'
  ...

src/channels/feishu.ts  (自注册)
  function createAdapter(): ChannelAdapter | null {
    const env = readEnvFile([...]);
    if (!env.APP_ID) return null;       ← 没凭证就不注册
    return { name, setup, deliver, ... };
  }
  registerChannelAdapter('feishu', { factory: createAdapter });  ← 侧效应注册
接口方法职责谁调用
setup(config)建立连接，注册回调Host 启动时
deliver(platformId, threadId, msg)发送消息到平台Delivery poll
setTyping(platformId, threadId)显示"正在输入"Typing refresh
teardown()断开连接Host 关闭时
isConnected()连接状态Health check

setup() 接收的 ChannelSetup 包含四个回调：

回调触发时机数据
onInbound(platformId, threadId, message)收到用户消息消息内容 + 元数据
onMetadata(platformId, name, isGroup)频道信息发现群/私聊标记
onInboundEvent(event)按钮点击等事件questionId + 选项
onAction(questionId, option, userId)用户操作交互式卡片
04 / 消息路由

一条消息的旅程

1

平台消息到达

Channel Adapter 调用 onInbound()，生成 InboundEvent

{ channelType: 'feishu', platformId: 'oc_group_abc', message: { content: '...', isMention: true } }
2

查找或创建 MessagingGroup

以 (channel_type, platform_id) 为 key 查找。群聊中 @bot 才自动创建，普通消息静默忽略

3

Sender Resolution

权限模块解析发送者身份（可选，无模块时 userId=null）

4

Fan-out 到所有 Wired Agent

一个群可以接入多个 agent，每个独立评估是否 engage

5

Engage Mode 评估

三种模式决定 agent 是否响应：

模式触发条件
pattern正则匹配消息文本（. = 永远响应）
mention必须 @bot
mention-sticky@bot 或已有活跃 session 的线程
6

Access Gate + Sender Scope

权限检查（无模块时默认 allow-all）

7

Resolve / Create Session

根据 session_mode 确定会话：

Session ModeSession Key说明
shared(agent, messaging_group)同群共享一个 session
per-thread(agent, messaging_group, thread)每个线程独立 session
agent-shared(agent_group)跨频道共享一个 session
8

写入 inbound.db + 唤醒容器

消息存入 messages_in 表，触发 wakeContainer()

MessagingGroup 确定规则

由 (channel_type, platform_id) 唯一确定。飞书群聊的 platform_id = 群 chat_id（所有人共享）；飞书私聊的 platform_id = P2P chat_id（每人独立）。不同 channel_type 天然隔离。

05 / 会话与上下文

Session 的隔离与共享

Session 是 NanoClaw 中上下文隔离的最小单位。Session key 不包含 user_id——它是按对话划分的，不是按用户划分的。

飞书群聊 (oc_group_abc)

  张三: "帮我查天气"  ──┐
  李四: "那上海呢"    ──┤── 同一个 messaging_group
  王五: "谢谢"       ──┘
         │
         ▼
  Session A (shared 模式)
  张三、李四、王五的消息都进同一个 session
  Claude 看到所有人的对话 → 上下文共享

────────────────────────────────────────

飞书私聊

  张三私聊 → platform_id=oc_dm_zhang → Session B (只有张三的对话)
  李四私聊 → platform_id=oc_dm_li   → Session C (只有李四的对话)
  两人各自独立，互不影响

────────────────────────────────────────

跨频道

  (feishu, oc_xxx) → MessagingGroup A → Session X
  (discord, chan-1) → MessagingGroup B → Session Y
  完全隔离，即使 platform_id 碰巧一样
群聊中所有人共享上下文

在飞书群里，所有人的消息进入同一个 session。Claude 看到的是完整的群聊记录，不区分说话人（sender 信息只是消息里的一个字段）。如果你需要"每个人有独立上下文"，应该用私聊。

06 / 数据库设计

三个数据库，各司其职

NanoClaw 有三类数据库文件，它们不在同一个 SQLite 里，没有外键关联。关系全靠代码逻辑串起来。

🗃 nanoclaw.db
全局唯一，进程级
sessions
会话记录 — id, agent_group_id, messaging_group_id, thread_id, status
messaging_groups
频道对话 — (channel_type, platform_id) 唯一
agent_groups
Agent 定义 — id, name, folder
messaging_group_agents
接线关系 — engage_mode, session_mode, priority
users / user_roles
用户与权限
✎ inbound.db
每 session 一个，Host 写
messages_in
用户消息 — seq(偶数), status, trigger, content
delivered
已投递记录 — 避免重复投递 agent 回复
destinations
消息目标 — agent 可发给谁
session_routing
默认回复地址 — channel_type + platform_id
💾 outbound.db
每 session 一个，Container 写
messages_out
Agent 回复 — seq(奇数), platform_id, content
processing_ack
处理确认 — pending→processing→completed
session_state
SDK session ID — continuation:claude = abc-123
container_state
当前执行的工具 — Host 用来判断是否卡住
为什么分两个 DB？

跨 VirtioFS mount 只能一方写 SQLite。如果 Host 和 Container 同时写同一个 DB，WAL 模式的 mmapped -shm 不会跨 mount 刷新，Container 会静默丢失新消息。所以用 journal_mode=DELETE + 两个文件，Host 写 inbound.db，Container 写 outbound.db，各写各的。

seq 偶数/奇数设计

messages_in.seq  = 偶数 (2, 4, 6, 8, ...)  ← Host 写
messages_out.seq = 奇数  (1, 3, 5, 7, ...)  ← Container 写

为什么？当 agent 要"编辑消息 #5"时，奇偶分立确保不会把 outbound 的 seq=5
误认为是 inbound 的 seq=5。两个命名空间，永不碰撞。

08 / 消息生命周期

一条消息的完整旅程

以用户在飞书群说"帮我查天气"为例，完整追踪数据在每个表之间的流转：

①

Host → inbound.db / messages_in

用户消息写入，status=pending，trigger=1（唤醒 agent）

id=om_001:ag-1 | seq=2 | content='{"text":"帮我查天气"}'
②

Host → inbound.db / session_routing

告诉 Container 回复往哪发

channel_type=feishu | platform_id=oc_group_abc
③

Container 读 messages_in → 写 outbound.db / processing_ack

标记正在处理，避免重复消费

message_id=om_001:ag-1 | status=processing
④

Container → outbound.db / session_state

SDK 返回 session_id，立即持久化（防崩溃丢失）

key=continuation:claude | value=abc-123-def
⑤

Container → outbound.db / messages_out

Agent 回复写入

id=out-001 | seq=1 | content='{"text":"北京晴，22°C"}'
⑥

Container → outbound.db / processing_ack

更新状态为 completed

message_id=om_001:ag-1 | status=completed
⑦

Host 读 messages_out → 通过 Adapter 投递

Delivery poll 读到新回复，调 Feishu adapter 发送

⑧

Host → inbound.db / delivered

记录已投递，防止重复发送

message_out_id=out-001 | status=delivered
⑨

Host 同步 processing_ack → inbound.db / messages_in

将 Container 的 completed 状态同步回来

id=om_001:ag-1 | status=completed
09 / SDK 会话续接

DB 与 Claude SDK 的联系

inbound.db / outbound.db 和 Claude SDK 的对话上下文是完全解耦的两层。唯一联系点是 outbound.db 里的一个 key：

outbound.db / session_state 表

  key = "continuation:claude"    value = "abc-123-def"
            │
            │  唯一的桥梁
            │
   ┌────────┴────────┐
   │                  │
   ▼                  ▼
DB 层              SDK 层
消息管道            对话续接
传输 & 记录        语义 & 上下文
   │                  │
   │  互相不知道       │  SDK 自己管理
   │  对方的内容       │  .jsonl transcript
   │                  │
   └──── poll loop ───┘
       读 DB → 推给 SDK
       SDK 返回 ID → 存回 DB

续接流程

第一次消息到达，continuation = undefined

sdkQuery({ prompt, options: { resume: undefined } }) → 全新对话

SDK 创建新 session，返回 session_id

init 事件 → setContinuation("claude", "abc-123")

session_id 立即写入 outbound.db，容器崩溃也不丢

同 session 内后续消息到达

continuation = "abc-123" (从 session_state 读取)
sdkQuery({ prompt, options: { resume: "abc-123" } })

SDK 找到 .jsonl transcript，恢复之前的对话上下文，Claude "记得"之前聊过什么

容器销毁后重建

outbound.db 在宿主机持久目录，不在容器内部，所以：

  1. 新容器挂载同一个 session 目录
  2. 从 outbound.db 读出 continuation:claude = abc-123
  3. SDK 用 resume: "abc-123" 恢复对话
  4. Claude 记得之前的所有上下文

用户发送 /clear 命令

continuation = undefined
clearContinuation("claude") → 删除 session_state 里的 key

下次消息到达时 SDK 不传 resume → 全新对话。但 inbound.db / outbound.db 的历史消息还在，只是 Claude 看不到了。

Agent 能访问你的文件吗

Agent 运行在 Docker 容器里，通过 volume mount 访问宿主机文件。默认只能操作自己的工作区。

Channel 自注册机制

NanoClaw 的 Channel 用自注册模式加载——import 即注册，不需要手动维护列表。

src/index.ts
  │
  └─ import './channels/index.js'     ← 启动时 import 触发链
       │
       ├─ import './cli.js'           ← CLI channel 自注册
       │    registerChannelAdapter('cli', { factory: createCliAdapter })
       │
       ├─ import './feishu.js'        ← 飞书 channel 自注册
       │    registerChannelAdapter('feishu', { factory: createAdapter })
       │
       ├─ import './telegram.js'      ← 加了 Telegram 就多一行
       │    registerChannelAdapter('telegram', { ... })
      ...
       │
       ▼
  initChannelAdapters()
    遍历 registry → 调用每个 factory()
    → 返回非 null 的 adapter → 调用 adapter.setup()
    → 返回 null（没配凭证）→ 跳过
加一个新 Channel 就是两步

① 把 channel 模块文件放到 src/channels/
② 在 src/channels/index.ts 加一行 import './xxx.js'
如果该 channel 的环境变量没配（如 FEISHU_APP_ID 为空），factory 返回 null，该 channel 自动跳过，不会报错。

14 / OneCLI 凭证网关

OneCLI 如何把 Key 注入容器

上一节提到容器永远看不到真实 API Key。OneCLI 是 NanoClaw 的默认凭证方案，通过 HTTPS MITM 代理在请求发出时动态注入密钥。整个过程中，真实密钥只存在 OneCLI Gateway 的加密 Vault 里。

三个角色

🔒

OneCLI Gateway

服务端，跑在 Docker 容器里。:10254 是管理面板，:10255 是 MITM 代理网关。内含 Secrets 密钥库、Agents 注册表、策略 Rules。

💻

onecli CLI

管理客户端，装在 ~/.local/bin/onecli。用来创建/查看 secrets、agents、rules。不影响容器运行，容器里没有这个二进制。

🐳

容器内 env

被代理方。只知道代理地址 + placeholder token。所有 HTTPS 流量自动走代理，不知道真实密钥。

架构全景

             OneCLI 凭证注入架构
══════════════════════════════════════════════════════════════

本机 (macOS / Linux)

  onecli CLI                    OneCLI Gateway (Docker)
  ~/.local/bin/onecli           :10254 管理面板 / API
       │                        :10255 代理网关
       │  REST API              ┌──────────────────────┐
       └───────────────────────►│  Secrets 密钥库       │
                                │  Agents 注册表        │

代理行为：所有 HTTPS 流量都走代理

容器的 HTTPS_PROXY 是全局设置，意味着所有 HTTPS 请求都会经过 Gateway。但代理行为取决于是否有匹配的 secret：

请求目标代理行为举例
匹配 secret 拦截 → 替换/注入认证 header → 转发 open.bigmodel.cn → 注入 Authorization: Bearer <real-key>
无匹配 直接透传，不做任何修改 github.com、npmjs.org 等
被 rule 拦截 阻止或限速 block / rate_limit 规则
安全边界

代理只改写 HTTP header，不修改请求体或响应体。对于没有匹配 secret 的请求，Gateway 是纯粹的透明代理——不读、不改、不记录。

完整流程（智谱 API 场景）

1. Host 启动容器时
  container-runner.ts:
    ① onecli.ensureAgent({ name, identifier })     ← 确保 agent 注册到 Vault
    ② onecli.applyContainerConfig(args)             ← 注入 HTTPS_PROXY + CA 证书
    ③ providers/claude.ts 检测 .env:                ← 注入 base URL + placeholder
       ANTHROPIC_BASE_URL  = https://open.bigmodel.cn/api/anthropic
       ANTHROPIC_AUTH_TOKEN = placeholder

2. 容器内 Claude SDK 发请求
  POST https://open.bigmodel.cn/api/anthropic/v1/messages
  Authorization: Bearer placeholder                 ← 假 token

  ┌────────────────────────────────────────────────────────┐
  │  因为 HTTPS_PROXY 设置，请求先到 OneCLI Gateway :10255  │
  └────────────────────────────────────────────────────────┘

3. Gateway 做 MITM
  ① 用 CA 证书解密 TLS                                ← 容器信任这个 CA (SSL_CERT_FILE)
  ② 读取 Host header: open.bigmodel.cn
  ③ 匹配 secret "Zhipu API Key"                       ← host_pattern 匹配
  ④ 替换 Authorization: Bearer dc3c4c...1DBM           ← 真实 key 替换 placeholder
  ⑤ 重新加密 TLS，转发到真实 open.bigmodel.cn

4. 响应回传
  open.bigmodel.cn → Gateway → 容器                    ← 透传，不做修改
  SDK 解析响应 → 写入 outbound.db
  Host delivery poll → 飞书 adapter → 用户收到回复 🎉

容器内关键环境变量

环境变量来源作用
HTTPS_PROXY OneCLI SDK applyContainerConfig 所有 HTTPS 请求走代理 (host.docker.internal:10255)
SSL_CERT_FILE OneCLI SDK 信任 Gateway CA，否则 MITM TLS 验证失败
NODE_EXTRA_CA_CERTS OneCLI SDK Node.js 额外信任的 CA 证书
NODE_USE_ENV_PROXY OneCLI SDK 让 Node.js 读取 HTTPS_PROXY 环境变量
ANTHROPIC_BASE_URL src/providers/claude.ts 读 .env SDK 请求发到智谱而非 api.anthropic.com
ANTHROPIC_AUTH_TOKEN src/providers/claude.ts 值为 placeholder，代理替换为真实 key

Secret 类型与注入方式

Secret 类型注入方式适用场景
anthropic 注入 x-api-key header Anthropic 官方 API (api.anthropic.com)
generic 自定义 header + value 模板 智谱、OpenRouter 等兼容 API，OAuth token
智谱 API 配置要点

智谱等兼容 API 使用 generic 类型 secret：

① host pattern = open.bigmodel.cn

② header name = Authorization

③ value format = Bearer {value}

④ .env 里设置 ANTHROPIC_BASE_URL=https://open.bigmodel.cn/api/anthropic

Gateway 匹配到请求后，将 Authorization: Bearer placeholder 替换为 Authorization: Bearer <真实 key>。
